## [2026-06-27] Архитектура системы

**Решение:** Единый корень `C:\Users\user\Documents\WORKSPACE\` — одно место для базы знаний, разработок и агентов.

**Обоснование:** Контекст не должен теряться между сессиями. Все агенты читают один источник правды. Синхронизация через GitHub → VPS.

**Статус:** активно

---

## [2026-06-27 23:38 +05:00] Автоматическая синхронизация WORKSPACE

**Агент:** Codex

**Решение:** GitHub `origin/main` используется как центральный узел. Windows и будущий VPS синхронизируют свои клоны каждые 5 минут по схеме commit → fetch → fast-forward/merge → push, без принудительного перезаписывания истории.

**Обоснование:** Git хранит версии и позволяет восстановление после поломки устройства. Сохранение конфликтов вместо `reset --hard` защищает одновременные изменения с нескольких машин.

**Статус:** частично выполнено — GitHub подключён, Windows-автоматизация подготовлена; VPS подключается после получения адреса и SSH-доступа.

---

## [2026-06-27 23:29 +05:00] Протокол контекста и памяти агентов

**Агент:** Codex

**Решение:** `AGENTS.md` назначен единым протоколом всех агентов, а `memory/STATE.md` — компактной горячей памятью. Детали остаются в проектных файлах, решениях, логах и сессиях; каждая запись подписывается агентом и системным временем.

**Обоснование:** единая короткая точка входа снижает расход токенов и риск потери контекста, а разделение горячей и долговременной памяти предотвращает дублирование базы.

**Статус:** активно

---

## [2026-06-27] Структура памяти агентов

**Решение:** Агенты пишут в `memory/log.md` (хронология) и `memory/sessions/ДАТА_агент.md` (детали). Решения — в `memory/decisions.md`.

**Обоснование:** Долгосрочная память через файлы, краткосрочная — через сессионные файлы. Новый агент читает log.md + нужный продукт README.md и сразу в контексте.

**Статус:** активно
